    - rejectMember calls auth.admin.deleteUser, sends rejection email via Resend, optionally inserts into blocked_emails
    - addMember creates auth user via auth.admin.createUser, creates members row with custom starting_points and approval_status 'approved'
    - removeMember calls auth.admin.deleteUser (cascades to members via FK)
    - updateMemberEmail updates email in both auth.users and members table
    - setMemberStartingPoints updates starting_points column in members table
    - setSecurityQuestion hashes the answer and upserts into admin_security_questions for the current admin
    - getSecurityQuestion returns the question text (not answer) for a given admin_user_id
    - verifySecurityAnswer compares provided answer hash against stored hash for given admin
    - resetOtherAdminEmail verifies the other admin's security answer, then updates their auth email
    - All admin actions verify the caller is an admin before proceeding

    1. Create `src/lib/validators/admin.ts`:

       - `updateEmailSchema`: Zod object with `member_id` (string, uuid), `new_email` (string, email)
       - `securityQuestionSchema`: Zod object with `question` (string, 5-200 chars), `answer` (string, 2-100 chars, trimmed, lowercased for case-insensitive matching)
       - `adminRecoverySchema`: Zod object with `target_admin_email` (string, email — the admin whose email is being reset), `security_answer` (string), `new_email` (string, email)
       - Export all schemas and inferred types

    2. Create `src/actions/admin/auth.ts`:
       - `'use server'` directive
       - `adminLogin(formData: FormData)`:
         a. Validate with adminLoginSchema
         b. Check email is one of ADMIN_EMAIL_GEORGE or ADMIN_EMAIL_DAVE (hardcoded check as secondary guard per research)
         c. Call `supabase.auth.signInWithPassword({ email, password })`
         d. On success, redirect to `/admin`

    3. Create `src/actions/admin/members.ts`:
       - `'use server'` directive
       - All actions use `createAdminClient()` (service_role) for user management
       - All actions verify the current user is admin before proceeding (call createServerSupabaseClient().auth.getUser() and check app_metadata.role)

       - `approveMember(memberId: string)`:
         a. Fetch member from members table
         b. Call `supabaseAdmin.auth.admin.inviteUserByEmail(email, { redirectTo: APP_URL + '/auth/callback?next=/dashboard' })`
         c. Update members: `approval_status = 'approved'`, `approved_at = now()`, `approved_by = currentAdminId`
         d. Return `{ success: true }`

       - `rejectMember(memberId: string, blockEmail: boolean)`:
         a. Fetch member to get user_id and email
         b. Delete auth user: `supabaseAdmin.auth.admin.deleteUser(userId)`
         c. Send rejection email via Resend (from: "George's Predictor", subject: "Registration Update", body: friendly message explaining their registration was not approved)
         d. If blockEmail: insert into blocked_emails table
         e. Return `{ success: true }`

       - `addMember(formData: FormData)`:
         a. Validate with addMemberSchema
         b. Create auth user: `supabaseAdmin.auth.admin.createUser({ email, email_confirm: true })`
         c. Update members row (created by trigger): set `approval_status = 'approved'`, `starting_points`, `display_name`
         d. Send welcome magic link via `supabaseAdmin.auth.admin.inviteUserByEmail(email, { redirectTo: ... })`
         e. Return `{ success: true }`

       - `removeMember(memberId: string)`:
         a. Fetch member to get user_id
         b. Delete auth user: `supabaseAdmin.auth.admin.deleteUser(userId)` — cascades via FK
         c. Return `{ success: true }`

       - `updateMemberEmail(memberId: string, newEmail: string)`:
         a. Validate email format
         b. Fetch member to get user_id
         c. Update auth user: `supabaseAdmin.auth.admin.updateUserById(userId, { email: newEmail })`
         d. Update members table: `email = newEmail`
         e. Return `{ success: true }`

       - `setMemberStartingPoints(memberId: string, points: number)`:
         a. Validate points >= 0
         b. Update members table: `starting_points = points`
         c. Return `{ success: true }`

    4. Create `src/actions/admin/recovery.ts`:
       - `'use server'` directive
       - All actions verify caller is admin before proceeding

       - `setSecurityQuestion(formData: FormData)`:
         a. Validate with securityQuestionSchema
         b. Get current admin user via createServerSupabaseClient().auth.getUser()
         c. Hash the answer (use a simple bcrypt-like approach: lowercase + trim the answer, then hash with Web Crypto API SHA-256 — no bcrypt dependency needed since these are personal questions, not passwords)
         d. Upsert into admin_security_questions: set question, answer_hash, updated_at = now() where admin_user_id = current user
         e. Return `{ success: true }`

       - `getSecurityQuestion(adminUserId: string)`:
         a. Verify caller is admin
         b. Query admin_security_questions where admin_user_id = adminUserId
         c. Return `{ question: string }` or `{ error: "No security question set" }`
         d. NEVER return the answer_hash

       - `verifySecurityAnswer(adminUserId: string, answer: string)`:
         a. Verify caller is admin
         b. Fetch the answer_hash for the target admin
         c. Hash the provided answer (same method as setSecurityQuestion: lowercase, trim, SHA-256)
         d. Compare hashes
         e. Return `{ verified: true }` or `{ verified: false }`

       - `resetOtherAdminEmail(formData: FormData)`:
         a. Validate with adminRecoverySchema
         b. Verify caller is admin and is NOT the target admin (can't reset your own email this way)
         c. Look up the target admin's user_id from auth.users by their current email
         d. Call verifySecurityAnswer for the target admin with the provided answer
         e. If verified: update auth user email via `supabaseAdmin.auth.admin.updateUserById(targetUserId, { email: newEmail })`
         f. Return `{ success: true }` or `{ error: "Security answer incorrect" }`
         g. Per CONTEXT.md locked decision: "Either admin can reset the other's account if email access is lost"

       - Test resetOtherAdminEmail: verifies security answer then updates email; rejects if answer wrong
