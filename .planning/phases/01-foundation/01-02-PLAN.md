    1. Create `src/lib/validators/auth.ts`:
       - `signupSchema`: Zod object with `display_name` (string, 1-50 chars, trimmed), `email` (string, email format, lowercase), `is_new_member` (boolean), `email_opt_in` (boolean, default true)
       - `loginSchema`: Zod object with `email` (string, email format, lowercase)

    2. Create `src/lib/email.ts`:
       - Import `Resend` from `resend`
       - Create a `resend` instance: `new Resend(process.env.RESEND_API_KEY)`
       - Export `sendEmail({ to, subject, html })` — generic wrapper around `resend.emails.send()` with `from: "George's Predictor <onboarding@resend.dev>"` (use onboarding@resend.dev for development; production will use a verified domain). Gracefully handles errors — logs and returns `{ error }` rather than throwing (email failure should not break signup).
       - Export `sendAdminSignupNotification({ displayName, email })` — sends an email to `process.env.ADMIN_EMAIL_GEORGE` with:
         - Subject: `"New signup waiting for approval: {displayName}"`
         - HTML body: A simple, clear email saying "{displayName} ({email}) has just signed up and is waiting for your approval. Log in to the admin dashboard to review."
         - Include a link to the admin dashboard: `${process.env.NEXT_PUBLIC_APP_URL}/admin/members?filter=pending`
       - Per CONTEXT.md locked decision: "George gets email notification AND admin dashboard alert for each new signup" — the DB trigger handles the dashboard alert, this handles the email leg.

    3. Create `src/actions/auth.ts`:
       - `'use server'` directive at top
       - `signUpMember(formData: FormData)`:
         a. Parse and validate with signupSchema
         b. Check blocked_emails table — if email is blocked, return `{ error: "This email address cannot be used for registration" }`
         c. Call `supabase.auth.signInWithOtp({ email, options: { shouldCreateUser: true, data: { display_name, email_opt_in } } })`
         d. Do NOT set emailRedirectTo — the initial OTP email is just a confirmation receipt, not the login mechanism. George sends the real magic link on approval.
         e. On successful signInWithOtp: call `sendAdminSignupNotification({ displayName: display_name, email })` to notify George via email. Do NOT await or block on this — fire and forget. If email fails, the signup still succeeds (the admin dashboard notification from the DB trigger is the backup).
         f. Return `{ success: true }` or `{ error: message }`
       - `requestMagicLink(formData: FormData)`:
         a. Parse and validate with loginSchema
         b. Call `supabase.auth.signInWithOtp({ email, options: { shouldCreateUser: false, emailRedirectTo: process.env.NEXT_PUBLIC_APP_URL + '/auth/callback?next=/dashboard' } })`
         c. Return `{ success: true }` or `{ error: message }`

    4. Create `tests/actions/auth.test.ts`:
       - Mock Supabase client (use helpers from tests/setup.ts)
       - Mock Resend (from tests/setup.ts)
       - Test all behaviors listed above
       - Verify signInWithOtp is called with correct parameters
       - Verify blocked email check queries blocked_emails table
       - Verify sendAdminSignupNotification is called after successful signup with correct displayName and email
       - Verify email failure does not cause signUpMember to fail (graceful degradation)

    IMPORTANT per user decision: No passwords for members — magic link only. The signInWithOtp with shouldCreateUser:true is the signup mechanism. The initial OTP email serves as "registration received" confirmation — members should NOT click it to log in (George's approval magic link is the actual login).

    IMPORTANT per locked decision: "George gets email notification AND admin dashboard alert for each new signup." The DB trigger (Plan 01-01) writes the admin_notifications row (dashboard alert). This plan's signUpMember action sends the Resend email (email notification). Both legs are covered.
